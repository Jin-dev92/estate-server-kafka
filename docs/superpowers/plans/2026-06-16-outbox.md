# Transactional Outbox 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 도메인 변경과 도메인 이벤트(outbox 행) INSERT를 한 DB 트랜잭션으로 커밋해 dual-write 이벤트 유실을 제거하고, 별도 outbox-relay 워커가 폴링으로 Kafka에 발행한다.

**Architecture:** 새 `TransactionRunner`/`OutboxStore` 포트로 board·membership 4개 use case가 `txRunner.run(tx => { repo.create(.., tx); outbox.add(event, tx) })` 형태로 전환한다. `outbox-relay` 워커가 트랜잭션 안에서 `SELECT … FOR UPDATE SKIP LOCKED`로 PENDING을 잠그며 가져와 Kafka emit → PUBLISHED 마킹한다. 소비자 멱등(`eventId @unique`)이 at-least-once 중복을 흡수한다.

**Tech Stack:** NestJS, Prisma(interactive `$transaction`, `$queryRaw` raw SQL), Kafka(`@nestjs/microservices` ClientKafka), Jest.

> **설계 스펙:** [docs/superpowers/specs/2026-06-16-outbox-design.md](../specs/2026-06-16-outbox-design.md)

## 핵심 설계 메모 (구현 전 반드시 읽기)

- **PrismaService**는 `PrismaClient`를 extends → `$transaction`/`$queryRaw` 사용 가능. interactive transaction 콜백은 `Prisma.TransactionClient` 타입을 준다(모델 delegate + `$queryRaw` 사용 가능, `$transaction` 자체는 불가).
- **tx 전파:** repository 메서드에 선택적 `tx?: TransactionClient` 인자를 추가. 구현은 `const db = tx ?? this.prisma; db.post.create(...)`. tx 없으면 기존 동작 동일(호출부 무변경).
- **SKIP LOCKED 정확성:** `SELECT … FOR UPDATE SKIP LOCKED`의 잠금은 **트랜잭션이 살아있는 동안만** 유효하다. 따라서 relay 1틱 전체(fetch→emit→mark)를 **하나의 `txRunner.run`** 으로 감싼다. 그래야 여러 relay 인스턴스가 같은 행을 동시에 잡지 않는다. emit이 트랜잭션 안이지만 배치가 작아 허용(학습 트레이드오프, 주석으로 명시).
- **토픽 매핑 단일 출처:** 현재 `kafka-event.publisher.ts`의 `TOPIC_BY_EVENT`를 `src/events/event-type.enum.ts` 옆 공유 헬퍼 `topicForEvent(eventType)`로 추출해 publisher와 outbox.add가 공유한다.
