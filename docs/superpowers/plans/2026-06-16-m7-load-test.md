# M7: k6 API 부하테스트 구현 계획

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 성격이 다른 대표 엔드포인트 4개에 k6 부하 시나리오(smoke/load 프로파일 + thresholds)를 만들고, 멱등 시드로 데이터를 준비해 성능 baseline(p95·RPS·에러율)을 측정·기록한다.

**Architecture:** `load/` 폴더에 k6(JS) 스크립트를 둔다. `config.js`가 env로 프로파일·threshold를 분기하고, `lib/auth.js`가 `setup()`에서 시드 유저로 로그인해 토큰·buildingId를 확보한다. 시나리오 4개(read/create/login/rate-limit)가 각자 thresholds로 합격 판정한다. `prisma/seed-load.ts`가 부하용 OWNER·건물·글을 멱등 생성한다.

**Tech Stack:** k6(외부 도구, JS ES module), Prisma(seed, ts-node), 기존 NestJS API.

> **설계 스펙:** [docs/superpowers/specs/2026-06-16-m7-load-test-design.md](../specs/2026-06-16-m7-load-test-design.md)

- 인프라: `docker compose up -d`(PG·Redis·Kafka). 앱: `node dist/main.js`.
- 컨벤션: 커밋 `[M7]{타입}: {한글}` + 트레일러 `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`. k6 스크립트는 jest 대상 아님(ESLint 대상에서도 `load/`는 제외 — Task 1에서 처리).

## 확인된 기존 사실 (구현 근거)

- 비밀번호 해시: `BcryptPasswordHasher`, **bcrypt rounds=10**. 시드도 `bcrypt.hash(pw, 10)`으로 저장해야 로그인 가능.
- 로그인 응답: `POST /auth/login` → `{ "accessToken": string }`.
- 게시글 목록(멤버 전용): `GET /buildings/:buildingId/posts` (Bearer 필요).
- 글 작성(멤버 전용): `POST /buildings/:buildingId/posts` body `{ category, title, content }`.
- `signup`은 role을 받지 않음 → seed에서 `role:'OWNER'` 직접 지정. OWNER는 자기 건물의 멤버로 인정됨(`PrismaMembershipChecker`).
- 모델: `User{ id,email@unique,passwordHash,name,role,... }`, `Building{ id,ownerId,name,address,... }`, `Post{ buildingId,authorId,category,title,content,... }`.
- rate limit env: `RATE_LIMIT_USER_MAX`, `RATE_LIMIT_IP_MAX`, `RATE_LIMIT_WINDOW_SEC`(M6).
