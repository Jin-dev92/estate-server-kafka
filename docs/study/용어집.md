# estate-server 용어집 (Glossary)

> 마일스톤 문서(설계 스펙·구현 계획·[학습 노트](마일스톤-학습-노트.md)·README)에 흩어진 용어를 한 곳에 모았다.
> 각 항목은 **뜻 → (우리 프로젝트에서의 쓰임/마일스톤)** 순서다. 더 깊은 맥락·트레이드오프·스스로 점검 문항은 [학습 노트](마일스톤-학습-노트.md)를 본다.
>
> 처음 보는 개념은 채팅 설명에 그치지 않고 이 문서·학습 노트에 적어 둔다(미래의 나·리뷰어를 위해).

---

## 1. 아키텍처 · 설계 (DDD)

- **DDD (Domain-Driven Design, 도메인 주도 설계):** 코드를 기술이 아니라 **업무 도메인 중심**으로 설계하는 방법론. *우리:* 전 마일스톤의 골격.
- **바운디드 컨텍스트 (Bounded Context):** 한 도메인 개념이 일관된 의미를 갖는 경계. *우리:* `auth·property·board·chat·notification·audit` 각각이 하나의 컨텍스트 = NestJS 모듈.
- **레이어드 아키텍처 (단방향):** `interface → application → domain → infrastructure` 한 방향 의존. interface=컨트롤러/게이트웨이(전송만), application=유스케이스(흐름 조립), domain=순수 규칙·포트, infrastructure=포트 구현(어댑터). *우리:* 모든 컨텍스트 공통.
- **의존성 역전 (DIP):** 상위(도메인)가 하위(DB·Kafka)를 직접 import하지 않고, 도메인이 정의한 **포트(인터페이스)**에 하위 구현이 맞춘다. *우리:* `Symbol 토큰 포트`(예: `OUTBOX_STORE`)를 도메인이 알고 Prisma 구현은 모듈에서 주입 → 도메인이 Prisma/Redis/Kafka를 모름.
- **포트 & 어댑터 (Hexagonal):** 포트=도메인이 요구하는 인터페이스, 어댑터=그 인터페이스의 기술별 구현. *우리:* `OutboxStore`(포트) ↔ `PrismaOutboxStore`(어댑터).
- **애그리거트 (Aggregate):** 함께 변경되어야 하는 엔티티 묶음. "**애그리거트 경계 = 트랜잭션 경계 = 정합성 경계**"가 핵심 원칙.
- **도메인 이벤트 (Domain Event):** 도메인에서 "일어난 사실"(`PostCreated`, `MessageSent` 등). *우리:* 컨텍스트 간 결합을 낮추는 통신 수단, Kafka로 발행.
- **유비쿼터스 언어:** 코드·문서·대화에서 같은 도메인 용어를 일관되게 쓰는 것(건물·호실·입주 등).

## 2. 인증 · 인가 · 보안 (M0·M1·M6)

- **인증 (Authentication):** "**누구인가**"를 확인. *우리:* bcrypt 비밀번호 해시 + JWT 발급, Passport `jwt` 전략 검증.
- **인가 (Authorization):** "**무엇을 할 수 있는가**"를 확인. *우리:* RBAC + 리소스 소유권 **이중**.
- **JWT (JSON Web Token):** `header.payload.signature` 구조의 서명된 토큰. 서버가 상태를 안 가져도 서명만으로 검증된다(stateless).
- **bcrypt:** 비밀번호 해시 함수. **cost(rounds)**가 높을수록 느려져 무차별 대입에 강함(= CPU 바운드). *우리:* 로그인 p95 114ms로 가장 무거운 엔드포인트(M7).
- **RBAC (Role-Based Access Control):** 역할 기반 접근 제어(`@Roles(OWNER)` + `RolesGuard`).
- **리소스 소유권 검사:** "이 건물/방/글의 소유자·멤버인가"까지 확인. 역할만 보면 "OWNER A가 B의 건물을 수정"하는 우회가 뚫리므로 필수.
- **rate limit:** 일정 시간(window) 안의 요청 수를 제한해 초과분을 **429로 거부**하는 보호 정책. *우리:* userId+IP 이중, Redis 고정 윈도우 카운터(M6).
- **고정 윈도우 (fixed window):** 시간을 고정 칸으로 쪼개 칸마다 카운트(예: 10초당 200). 단순하지만 **경계에서 2배 버스트** 가능 → 슬라이딩 윈도우로 완화.
- **rate limit ≠ debounce/throttle:** debounce/throttle은 *클라이언트*가 호출을 조용히 줄이는 UX 기법, rate limit은 *서버*가 초과를 거부하는 보호. **≠ DDoS 방어**(분산 공격은 CDN/WAF/네트워크 계층의 일).

## 3. Prisma · 관계형 DB (M0·M1·M2.7)

- **Prisma:** TypeScript ORM. 스키마(`schema.prisma`)·마이그레이션·타입 안전 쿼리.
- **마이그레이션:** 스키마 변경을 SQL 파일로 형상관리. `migrate dev`(개발: 스키마→SQL 생성+적용) vs `migrate deploy`(운영: 생성된 것만 적용, 롤백 플랜 필수).
- **cuid:** 충돌 적은 문자열 PK 생성기(`@default(cuid())`). uuid 대안.
- **named relation:** 같은 모델을 두 FK가 가리킬 때 구분하는 관계 이름(예: ChatRoom의 owner/tenant 둘 다 User).
- **복합 인덱스 / 유니크:** `@@index([a,b])`(조회 가속) vs `@@unique([a,b])`(중복 방지 + 조회 가속). 컬럼 순서·카디널리티가 성능을 좌우.
- **soft delete (논리삭제):** 행을 지우지 않고 `deletedAt`만 채워 "지운 것처럼" 다룸. *우리:* 5개 엔티티에 적용, repository에서 `deletedAt: null` 필터를 **캡슐화**(도메인은 모름). 함정: `email @unique` 충돌.
- **cascade (연쇄 삭제):** 부모 삭제 시 자식 자동 삭제. *우리:* soft delete 도입 후 DB `onDelete: Cascade`가 무의미 → Post soft delete 시 자식 Comment를 **앱 레벨·같은 트랜잭션**으로 함께 처리.

## 4. Redis (M1·M2·M4·M5·M6)

- **TTL (Time To Live):** 키의 자동 만료 시간(`SET key val EX <초>`). *우리:* 초대코드 만료(M1).
- **read-through 캐시:** 읽을 때 캐시에 없으면 DB에서 읽어 캐시를 채우는 패턴. *우리:* 게시판(M2).
- **캐시 무효화 (invalidation):** 원본이 바뀌면 캐시를 버리는 것. *우리:* "쓰기 시 명시적 삭제 + 짧은 TTL 안전망" 이중 방어. 캐시의 본질적 난제.
- **pub/sub:** 발행(publish)/구독(subscribe) 메시징. *우리:* 단일 채널 발행 → 모든 인스턴스 구독 → 로컬 소켓에 emit(멀티 인스턴스 실시간 전달, M4·M5). 구독 전용 연결이 필요(`redis.duplicate()`).
- **capped list:** 최근 N개만 유지하는 리스트(`LPUSH` + `LTRIM 0 N-1`). *우리:* 채팅 최근 메시지 캐시(M4).
- **원자적 카운터:** `INCR`/`DEL`로 동시성에도 유실 없이 세기. *우리:* 미읽음 수(M5). **비정규화 캐시**라 eviction 시 유실 → DB COUNT 폴백이 정공법.
- **Lua 원자성:** 여러 Redis 명령을 한 스크립트로 묶어 원자 실행. *우리:* rate limit의 `INCR`+`EXPIRE`를 한 Lua로(분리 시 "INCR 후 크래시 → TTL 없는 영구 키" race 방지).

## 5. Kafka · 이벤트 드리븐 (M3·M4·M5)

- **producer / consumer:** 메시지 발행자 / 소비자. *우리:* main이 producer, 워커들이 consumer.
- **topic / partition:** 토픽=메시지 분류, 파티션=토픽을 쪼갠 병렬·순서 단위. *우리:* `chat-events`·`board-events`·`membership-events`.
- **consumer group:** 같은 group 안에선 파티션을 나눠 갖고(부하 분산), **다른 group은 같은 메시지를 각자 한 번씩** 받음(팬아웃).
- **팬아웃 (fan-out):** 이벤트 1건을 여러 소비자가 각자 한 번씩 받는 것. *우리:* persistence·notification·audit 세 group 독립 소비(M5).
- **파티션 키:** 같은 키는 같은 파티션으로 → **순서 보장**. *우리:* `roomId`로 방 내 메시지 순서 보장(M4).
- **at-least-once / at-most-once / exactly-once:** 전달 보장 등급. at-least-once="최소 한 번"(유실 없음·중복 가능). *우리:* at-least-once + **멱등 소비**로 중복을 흡수.
- **멱등 소비 (idempotent consumer):** 같은 메시지를 여러 번 받아도 결과가 한 번과 같게. *우리:* **앱 생성 ID를 유니크 키로** 두고 중복 INSERT 시 `P2002`(unique 위반) 무시.
- **offset:** consumer가 어디까지 읽었는지 위치. commit 시점에 따라 중복/유실이 갈림.
- **쓰기 버퍼:** 동기 INSERT 대신 Kafka에 흘려 **쓰기 스파이크를 흡수**. *우리:* 채팅 메시지(M4).
- **KRaft:** ZooKeeper 없이 Kafka 자체가 메타데이터를 관리하는 모드. *우리:* docker-compose의 cp-kafka.
- **토픽 사전생성:** 운영에선 `auto.create.topics.enable=false`(오타 토픽·파티션 미제어 방지) → 부팅 시 명시적으로 생성. 콜드스타트 race를 `ensureTopics()` await로 해결.

## 6. 정합성 · 신뢰성 (M3·M4·M5·Outbox·M9)

- **멱등성 (Idempotency):** 같은 작업을 여러 번 해도 결과가 한 번과 같음. 비동기 시스템의 기둥.
- **결과적 정합성 (Eventual Consistency):** 즉시 일관이 아니라 "잠시 후 일관". *우리:* "전달은 됐는데 DB엔 아직" 윈도우(채팅) → Outbox로 해소.
- **dual-write 문제:** DB 쓰기와 메시지 발행을 따로 하면, 그 사이 크래시 시 "DB는 썼는데 이벤트 유실". *우리:* Outbox로 제거.
- **Transactional Outbox:** 도메인 변경 + 이벤트(`OutboxEvent`) INSERT를 **한 트랜잭션**으로 커밋 → 유실 불가. 별도 **relay 워커**가 PENDING을 폴링해 Kafka 발행 후 PUBLISHED 마킹.
- **relay (릴레이) 워커:** outbox 행을 주기 폴링해 실제로 Kafka에 발행하는 별도 프로세스.
- **FOR UPDATE SKIP LOCKED:** 행을 잠그되 이미 잠긴 행은 건너뛰고 가져오는 SQL. *우리:* 멀티 relay가 같은 행을 중복 발행하지 않게(잠금은 트랜잭션 동안 유효).
- **DLQ (Dead Letter Queue, 데드 레터 큐):** "죽은 편지함"(수취인 불명 편지를 모으던 데서 유래). **반복 실패한 메시지를 정상 흐름에서 빼내 격리하는 곳**. *우리(M9):* 별도 큐 없이 outbox 행 `status`를 `FAILED`로 전환 = DLQ 역할.
- **poison message (독약 메시지):** 영구 오류로 **몇 번을 재시도해도 영원히 실패하는 메시지**. 정상 큐에 두면 자원·로그 낭비 + 뒤를 막음 → DLQ 격리 대상.
- **지수 백오프 (exponential backoff):** 재시도 간격을 실패할수록 2배씩(1→2→4→8s…) 늘리고 **cap(상한)**을 둠. 일시 장애엔 빠른 복구, 지속 장애엔 부하 절감. *우리(M9):* `computeBackoff = min(base*2^n, cap)`.
- **head-of-line blocking:** 맨 앞의 막힌 항목이 뒤 전체를 못 가게 막는 현상. *우리:* poison message가 배치 앞을 막을 위험 → DLQ로 해소.
- **best-effort:** "되면 좋고 안 돼도 치명적이지 않은" 처리. *우리:* 알림 WS 푸시(적재·카운터가 **진실 원천**이라 푸시 실패가 적재를 막지 않게 try/catch).
- **진실 원천 (source of truth):** 데이터의 정답이 되는 곳. *우리:* 미읽음은 DB가 진실, Redis 카운터는 캐시.

## 7. 에러 · API 계약 (M2.5·M2.6)

- **에러 envelope:** 에러 종류와 무관하게 **겉 JSON 구조를 항상 동일**하게 감싸는 표준 포장(`statusCode·code·message·path·timestamp`). *우리:* `AllExceptionsFilter`가 모든 4xx/5xx를 통일.
- **`code` 계약:** FE는 바뀔 수 있는 `message` 문구가 아니라 **안정적인 `code`**(`BOARD_POST_NOT_FOUND`)로 분기. 변하지 않는 약속.
- **ExceptionFilter:** NestJS의 전역 예외 변환기. 커스텀 예외 → envelope.
- **Swagger / OpenAPI:** API 명세 자동 문서화. *우리:* `@ApiTags/@ApiOperation/@ApiResponse`, enum은 `@ApiProperty({ enum, enumName })`로 명명 스키마 노출.

## 8. 성능 · 부하테스트 (k6, M7·M8)

- **k6:** JS로 시나리오를 쓰는 부하테스트 도구(자체 런타임).
- **p95 / p99 (백분위):** "100명 중 95(99)등의 응답시간". 평균은 **느린 꼬리를 숨기므로** 백분위로 본다.
- **RPS (Requests Per Second):** 초당 요청 수 = 부하의 세기.
- **VU (Virtual User):** 가상 사용자. **iteration**=시나리오 1회 실행.
- **closed (VU) 모델:** 응답을 기다렸다 다음 요청 → 느려지면 부하도 줄어 **backpressure가 숨음**(baseline에 적합).
- **open (arrival-rate) 모델:** "초당 X건" 도착률을 고정해 밀어붙임 → 적체·한계를 드러냄(**stress/spike에 필수**).
- **arrival-rate executor:** open 모델 실행기(`ramping-arrival-rate` 등). `preAllocatedVUs`/`maxVUs`로 in-flight 상한 지정.
- **smoke / load / stress / spike:** 정상성 확인 / baseline 측정 / 한계점 탐색 / 급증 충격·회복. 목적이 다른 부하 4종.
- **knee (무릎):** 부하를 올리다 p95가 갑자기 꺾여 치솟는 지점 = 한계점. *우리(M8):* DB 풀=1로 좁혀 p95 13.6→1734ms 폭증·throughput ~95 RPS 천장 관측.
- **병목 (bottleneck):** 한계에서 먼저 터지는 자원. *우리(M8):* DB 커넥션 풀(Prisma P2024 풀 타임아웃으로 확인).
- **backpressure:** 처리 속도를 못 따라가는 부하가 쌓이며 생기는 역압. open 모델이라야 드러남.
- **connection pool (커넥션 풀):** 앱이 DB와 미리 맺어 돌려쓰는 연결 묶음. 풀이 N이면 동시 쿼리 N개, 나머지는 대기. `connection_limit`·`pool_timeout`으로 조절.
- **threshold vs check:** threshold=합격/불합격 게이트(미달 시 k6 exit≠0 → CI red), check=개별 응답 검증(실패해도 계속).
- **think time:** 요청 사이 쉬는 시간(`sleep`). 넣으면 현실적 동시 사용자, 빼면 최대 처리량.
- **Little's law:** 동시성 = 처리량 × 지연. 용량 계획의 기본 식.
- **SLI / SLO / error budget:** 지표 / 목표치 / 허용 오차 예산. p95 SLO 등.

## 9. 테스트 (전 마일스톤)

- **TDD (Test-Driven Development):** 실패 테스트(RED) → 최소 구현(GREEN) → 정리(REFACTOR) 반복.
- **AAA 패턴:** Arrange(준비) - Act(실행) - Assert(검증), 블록 사이 빈 줄로 구분.
- **테스트 더블:** 가짜 의존. stub(고정 응답)/mock(호출 검증)/fake(간이 구현)/spy(실제+감시). *우리:* 외부 의존(DB·Redis·Kafka)은 mock, 유스케이스는 포트를 가짜로 주입해 격리.
- **격리 테스트:** 외부 의존을 끊고 단위만 검증. 포트(인터페이스) 덕에 가능.
