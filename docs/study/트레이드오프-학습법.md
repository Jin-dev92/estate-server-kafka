# 기술 용어·트레이드오프를 파악하는 법 — 학습 가이드 + 치트시트 + 셀프 퀴즈

> 이 문서는 "구현은 따라가는데 **기술 용어와 트레이드오프를 스스로 짚는 게 약하다**"는 자각에서 출발한 학습 도구다.
> estate-server 프로젝트(M0~M6 + Outbox)에서 실제로 내린 결정들을 재료로 삼는다.
> 배경: [README §5 설계 결정·트레이드오프](../../README.md) · [마일스톤 학습 노트](마일스톤-학습-노트.md)

---

## 0. 왜 어렵게 느껴지나 (재프레임)

트레이드오프·용어가 막막한 건 보통 **"정답을 외우려"** 하기 때문이다. 둘의 본질은 정답이 아니다.

- **용어 = "어떤 고통(문제)에 붙인 이름"**
  `Transactional Outbox`는 외울 단어가 아니라 *"dual-write 유실이라는 고통을 없애려고 만든 패턴의 이름"*이다.
  → 용어를 외우지 말고 **"이 단어가 없던 시절엔 뭐가 아팠나?"**를 먼저 떠올려라. 정의가 따라온다.

- **트레이드오프 = "축(axis) 위에서 한 쪽을 고른 것"**
  모든 결정엔 "무엇을 얻고 무엇을 잃었나"의 짝이 있다. **축을 모르면 트레이드오프가 안 보인다.**

## 1. 사고 도구 ① — 트레이드오프 "축" 8개

새 기술/설계를 만나면 단 하나를 물어라: **"이건 어느 축에서 어느 쪽을 고른 거지?"**
반복되는 축은 의외로 적다. 이 8개면 백엔드 결정의 대부분이 설명된다.

| # | 축 (A ↔ B) | 한 줄 의미 | 이 프로젝트의 예 |
|---|---|---|---|
| 1 | **체감 지연 ↔ 정합성** | 빨리 보여주기 vs 정확히 맞추기 | 채팅 실시간 전달(Redis) vs 영속화(Kafka) 분리 / Outbox 폴링 지연 vs 유실 제거 |
| 2 | **단순함 ↔ 정확함** | 구현·이해 쉬움 vs 엣지까지 정확 | rate limit 고정 윈도우(단순, 경계 버스트) vs 슬라이딩(정확, 복잡) |
| 3 | **유실 없음 ↔ 중복 없음** | 메시징의 근본 한계 | Outbox/Kafka = at-least-once (유실X·중복O → **멱등**으로 보완) |
| 4 | **결합도 ↓ ↔ 보일러플레이트 ↑** | 갈아끼우기 쉬움 vs 코드량 | DDD 레이어드 + 의존성 역전(포트/어댑터) |
| 5 | **읽기 최적화 ↔ 쓰기 비용/일관성** | 읽기 빠르게 vs 쓰기·정합 부담 | 게시판 read-through 캐시 + 명시적 무효화 |
| 6 | **가용성 ↔ 보호(안전)** | 일단 통과 vs 막아서 지킴 | rate limit **fail-closed**(Redis 죽으면 요청 거부) |
| 7 | **표현력/학습 표면 ↔ 모델 무게** | 의미를 살림 vs 단순함 | 도메인 3계층(건물→호실→입주) 유지 |
| 8 | **명시성 ↔ 마법(암묵)** | 눈에 보임·디버깅 쉬움 vs 적은 코드·숨음 | Outbox tx **명시 전파**(tx 인자) vs AsyncLocalStorage 암묵 전파 |

> **쓰는 법:** 새 라이브러리/패턴을 볼 때 위 표에서 "이건 몇 번 축이지?"를 찾는다. 안 맞으면 새 축을 하나 추가한다(축 목록을 늘려가는 게 곧 실력).

### 자주 나오는 "이름 붙은 트레이드오프"
- **CAP** — 분산에서 네트워크 분단(P) 시 일관성(C) vs 가용성(A) 중 택1.
- **at-least-once / at-most-once / exactly-once** — 메시지 전달 보장 3종. exactly-once는 비싸서, 보통 **at-least-once + 멱등**으로 흉내 낸다(우리 선택).
- **read-your-writes** — 내가 쓴 건 바로 보여야 한다는 일관성 요구(채팅 "전달은 됐는데 DB엔 아직" 윈도우가 이걸 약하게 깬다).
- **fail-open / fail-closed** — 의존성이 죽었을 때 통과시킬지(open) 막을지(closed). 보안은 보통 closed.

## 2. 사고 도구 ② — 결정마다 "3단 + 반대선택" 카드

README §5가 이미 이 구조다. 직접 만들 땐:

1. **무엇을?** (what) — 한 줄
2. **왜?** (why) — *어떤 문제를 풀려고*
3. **무엇을 포기?** (trade-off) — 잃은 것 + 그게 속한 **축**
4. **★ 반대로 했다면?** — 이게 핵심. *"반대 선택지의 장점과 그걸 안 고른 이유"*를 말할 수 있으면 그 트레이드오프를 **진짜** 이해한 것.

예시(결정 13, Outbox):
- **무엇을:** 도메인 변경 + outbox 행을 한 트랜잭션으로 커밋, relay가 폴링 발행.
- **왜:** dual-write(DB는 썼는데 이벤트 발행 실패) 유실을 없애려고.
- **포기:** 폴링 지연(축 1) + 중복 발행 가능(축 3, at-least-once).
- **반대로 했다면:** "그냥 after-commit 직접 발행"은 코드가 단순하지만 유실 창이 남는다. 학습/정합성 목적이라 Outbox를 골랐다. — *이걸 막힘없이 말할 수 있으면 합격.*

## 3. 실천 루틴 (이 프로젝트로)

1. **"스스로 점검" 질문에 소리 내어 답하기** (Feynman 기법)
   [학습 노트](마일스톤-학습-노트.md)의 각 `### 스스로 점검` 항목을 말로 설명. **막히는 질문 = 약한 지점.** 거기만 다시 판다.
2. **"더 팔 키워드" 역추적**
   노트마다 키워드 목록이 있다(예: `Saga`, `CDC`, `cache stampede`). 하나씩 *"이게 없으면 무슨 문제가 생기지?"*로 정의. 검색은 **"X vs Y"** 형태로("Outbox vs Saga", "fixed vs sliding window") — 비교 글이 트레이드오프를 가장 잘 드러낸다.
3. **"반대 선택" 5분 설명 녹음**
   우리 결정 하나를 골라 "왜 반대를 안 했는지"를 말로. 말이 막히면 글로도 못 쓴 것.
4. **새 PR마다 "배운 것 3줄"**
   이미 PR 본문에 넣고 있다. 그 3줄을 항상 *축 번호와 함께* 쓰면 축 감각이 는다.

## 4. 셀프 퀴즈 — "너라면 뭘 고르고 뭘 포기?"

아무것도 안 보고 답해보고, 그다음 README §5나 학습 노트와 대조한다. **정답을 맞히는 것보다 "어느 축의 문제인지"를 짚는 게 핵심.**

1. 게시판 글 목록을 캐시한다. **"쓰기 후 캐시 삭제" vs "쓰기 후 캐시 갱신(write-through)"** 중 무엇을, 왜? (힌트: 축 5)
2. 채팅 메시지를 "전달은 됐는데 DB엔 아직" 윈도우가 있다. 이게 **어느 일관성**을 깨고, 왜 학습 단계에선 허용했나? (축 1)
3. rate limit에서 **고정 윈도우** 대신 **슬라이딩 윈도우**를 골랐다면 무엇이 좋아지고 무엇이 나빠지나? (축 2)
4. Kafka를 at-least-once로 두고 **소비자 멱등**으로 보완했다. exactly-once를 직접 추구하지 않은 이유는? (축 3)
5. Redis가 죽었을 때 rate limit 가드는 요청을 **통과시킬까 막을까?** 우리 선택과 그 이유는? (축 6 / fail-open vs closed)
6. Outbox에서 트랜잭션을 **tx 인자 명시 전파**로 했다. **AsyncLocalStorage 암묵 전파**의 장점은 뭐고, 왜 안 골랐나? (축 8)
7. 알림에서 **외부 푸시(FCM)를 뺐다.** 무엇을 잃었고, 이벤트 드리븐 구조 덕분에 나중에 어떻게 싸게 추가할 수 있나? (축 7 / 확장성)
8. 도메인을 2계층이 아니라 **3계층(건물→호실→입주)**으로 뒀다. 모델이 무거워지는 대가로 무엇을 얻었나? (축 7)
9. Outbox는 "유실 없음"은 주는데 **"중복 없음"은 왜 못 주나?** 그 짝꿍은? (축 3)
10. 컨슈머를 **워커 프로세스로 분리**한 대가(프로세스 4개)와 이득(독립 배포·진짜 팬아웃)은? (축 4 + 운영)

> **채점 기준:** "A가 맞다"가 아니라 **"A는 ○○을 얻고 ○○을 잃는다, 반대 B는 그 반대다, 그래서 우리 상황에선 A"** 까지 말하면 통과.
